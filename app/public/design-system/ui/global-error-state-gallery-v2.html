Recovery action patterns — shows the decision tree for each error type. Defines which CTA appears, when to show support, when to auto-retry.
Recovery Action Patterns
Decision tree for error CTAs by type and audience
🌐 Network Error Recovery
Applied to: all audiences when offline or connection fails
1
Auto-retry once after 3 seconds silently
User sees "Reconnecting…" indicator
2
Show error state if auto-retry fails
CTA: "Try Again" button
3
If 3 manual retries fail, show support path
CTA: "Get Help" → support form
💥 Server Error Recovery
Applied to: 500/502/503 responses from WQ API
1
Show friendly error, never raw status code to child/parent
Child: "Something went wobbly" · Parent: "Something went wrong"
2
Teacher/Owner sees HTTP status code in subdued text
Never stack trace, never database errors
3
Log error to monitoring automatically
Sentry / DataDog event created
🔑 Session Expired Recovery
Applied to: all audiences when auth token expires
1
Save current page state to local storage before redirecting
User returns to same page after sign-in
2
Show session expired message, not auth error code
CTA: "Sign Back In" → auth screen
3
Child: show parent hand-off message instead
"Ask a grown-up to sign you back in"
🚫 Permission Denied Recovery
Applied to: wrong role access or expired subscription
1
Explain what type of account is needed, not why access was denied
"This is for teachers — you have a parent account"
2
Provide clear path back to the correct surface
CTA: "Go to My Dashboard"
3
Never tell users they were blocked or flagged
Neutral language only

Language Rules by Audience
ChildZero jargon, friendly, no blame, always Try Again
ParentPlain English, no HTTP codes, always retry path
TeacherInformative, HTTP status okay, no stack traces
OwnerFull error codes, timestamps, no stack traces
Auto-Retry Rules
Network errors: auto-retry 1x after 3s
Server errors: show error immediately, no auto-retry
Session expiry: redirect to sign-in after 5s countdown
Rate limits: show countdown to reset time
