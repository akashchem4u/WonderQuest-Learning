Child Safety Rules

The following words are prohibited in hint_text and explainer_text. The UI must surface an amber warning if any are detected (case-insensitive).

Prohibited Word Required Replacement Style Example
wrongTry again! 🌟"That's not quite right — try again! 🌟"
incorrectAlmost there! 🌟"Almost there! 🌟 Look at the picture for a clue."
mistakeGood effort! 💫"Good effort! 💫 Let's think about this together."
failedKeep going! 🚀"Keep going! 🚀 You're learning something new."
badYou've got this! ⭐"You've got this! ⭐ Try one more time."

Business Rules

  • 1Child Safety Gate: child_safety_approved must be true before status can transition to published. The API must enforce this server-side, not UI-only.
  • 2Prohibited Words: The save/submit API must reject hint_text or explainer_text containing: wrong, incorrect, mistake, failed, bad — returning HTTP 422 with field-level error messages.
  • 3No Child PII: Question content fields (question_text, hint_text, explainer_text, answer_options) must not contain child names, IDs, or any PII. Content team training must reinforce this.
  • 4Multiple Choice Minimum: multiple_choice questions must have at least 2 and at most 4 answer options, with exactly one marked as correct.
  • 5Audit Trail: All status transitions must be logged with reviewed_by (user ID), timestamp, and previous/new status. This log is retained for 7 years.
  • 6Band–Subject Validation: Every published question must have a non-null band and subject. The Quest engine uses these to personalise question delivery per child's learning profile.
