🔊 Voice Control Patterns

WonderQuest · v2 — Audio interaction patterns for child play experience

voice-control-patterns-v2.html — Voice interaction patterns for child play. Voice modes: (1) Auto-read (question read aloud automatically on load), (2) Tap-to-hear (child taps speaker icon to replay), (3) Voice answer (child speaks answer — optional, device-permitting). Audio: Web Speech API (TTS) for readout. Voice answer: SpeechRecognition API (optional feature, graceful fallback). No timer anywhere. Audio: LUFS target -18±2 dB. Voice actor: pre-recorded assets (not live TTS for MVP). TTS only as fallback.

Web Speech API — TTS Fallback

🗣️ SpeechSynthesis (TTS)

TTS is fallback only. Pre-recorded voice actor assets are used for MVP (LUFS target -18±2 dB). Rate 0.9 — slightly slower than default for child comprehension. 

const utterance = new SpeechSynthesisUtterance(questionText);
utterance.rate  = 0.9;   // slightly slower for children
utterance.pitch = 1.0;
utterance.lang  = 'en-US';
speechSynthesis.speak(utterance);

// Graceful fallback check
if (!('speechSynthesis' in window)) {
  // Audio unavailable — text is always visible, gameplay continues
  showFallbackText();
}
SpeechRecognition — Voice Answer (Optional Feature)

🎤 SpeechRecognition

Optional feature shipping in v2.7. Always gracefully falls back to tap-to-answer if recognition is unavailable or permission denied. Never blocks gameplay. 

const recognition = new (window.SpeechRecognition ||
  window.webkitSpeechRecognition)();

recognition.lang              = 'en-US';
recognition.interimResults    = false;
recognition.maxAlternatives   = 3;

recognition.onresult = (event) => {
  const transcript = event.results[0][0].transcript;
  showRecognizedText(transcript);  // → "I heard: Forty-two"
};

recognition.onerror = (event) => {
  // Show "Hmm, I didn't catch that" + fallback to tap
  handleVoiceError(event.error);
};

// Permission check before starting
navigator.permissions.query({ name: 'microphone' })
  .then(status => {
    if (status.state === 'denied') showTapFallback();
    else recognition.start();
  })
  .catch(() => showTapFallback());
Accessibility Rules

♿ Accessibility

  • 1 All audio content also available as text. Question text is always visible — never hidden or replaced by audio.
  • 2 Audio can be disabled via parent settings → audio_enabled: false in child profile. All animations and audio stop; text remains fully functional.
  • 3 prefers-reduced-motion: disables waveform and ring animations. Audio playback continues normally — decorative animation only.
  • 4 All waveform and ring animation elements carry aria-hidden="true" — purely decorative.
  • 5 Speaker button: aria-label="Read question aloud" (idle) / aria-label="Replay question" (complete state). Status changes announced via aria-live="polite".
  • 6 Answer buttons include aria-disabled="true" during readout. Keyboard and switch-access users hear "unavailable" during audio playback.
  • 7 Voice answer is always optional. A visible tap-to-answer path is always present. Microphone permission prompt includes a "No thanks — I'll tap instead" escape.
