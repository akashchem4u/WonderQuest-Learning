Database Schema
CREATE TABLE beta_issues ( id UUID PRIMARY KEY, ref_id TEXT UNIQUE, -- BTA-NNNN feedback_id UUID REFERENCES feedback(id), type ENUM('bug','suggestion','rating','observation'), priority ENUM('p0','p1','p2','p3') nullable, status ENUM('new','triaged','in_progress','resolved','wont_fix','deferred'), summary TEXT, tester_label TEXT, -- "Maple Ridge Elementary" or "Family Cohort B" (NOT personal name) cohort_type TEXT, beta_week INT, screen_context TEXT, assigned_team TEXT nullable, resolution_note TEXT nullable, resolved_at TIMESTAMPTZ nullable, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE beta_issue_notes ( id UUID, issue_id UUID, author_id UUID, body TEXT, created_at TIMESTAMPTZ );
Privacy Rule
tester_label stores school name or cohort label — NEVER an individual teacher or parent name unless the user explicitly opted in (opt_in_public_attribution = true in their beta settings). This field is the only tester-identifying data surfaced in this sheet.

Distinction from Production Feedback Queue
This sheet covers beta period issues only. The main owner feedback queue (owner-feedback-workbench) covers production feedback. Beta issues are tracked here from first submission through resolution and may be closed out at beta end or escalated to production tracking. Beta issues reference a feedback_id FK to the originating feedback record but are managed separately with beta-specific fields (cohort, beta_week, tester_label).
