Impact Summary

Audio playback was non-functional for learners accessing WonderQuest on iOS 17.3 devices. Approximately 340 learning sessions were interrupted during the window 02:14–04:47 UTC. Phonics and read-aloud activities were disproportionately affected. No child PII was exposed or logged during the incident window. A hotfix was deployed within the SLA window for P0 incidents.

Incident Timeline (UTC)

  1. 02:14 Alert fired — Automated monitoring detected elevated audio error rate on iOS clients; PagerDuty alert dispatched to on-call rotation.
  2. 02:19 On-call acknowledged — On-call engineer accepted the alert and began triage; war room channel opened.
  3. 02:51 Root cause identified — Safari 17 AudioContext policy change confirmed as the breaking factor; iOS lead escalated to platform team.
  4. 03:20 Hotfix drafted — User-gesture unlock wrapper added to AudioContext initialization; PR opened against main.
  5. 04:02 Staging validated — Automated regression suite and manual smoke test on iOS 17.3 simulator confirmed fix; QA sign-off recorded.
  6. 04:47 Production deploy — Hotfix merged and deployed; error rate returned to baseline; incident marked resolved.

Root Cause

Safari 17 (shipped with iOS 17.3) enforces a stricter AudioContext autoplay policy that suspends the AudioContext until a direct user gesture event is received. WonderQuest's audio engine initialized the AudioContext during module load — before any user interaction — which caused AudioContext.state to remain "suspended" indefinitely on iOS 17.3. This behavior was not covered by the existing browser compatibility matrix, and no iOS 17 device was included in the automated regression suite.

DB Schema

incidents {
  id                  UUID PRIMARY KEY,
  title               TEXT NOT NULL,
  severity            ENUM('P0', 'P1', 'P2') NOT NULL,
  status              ENUM('open', 'mitigated', 'resolved', 'postmortem_complete') NOT NULL,
  detected_at         TIMESTAMPTZ NOT NULL,
  mitigated_at        TIMESTAMPTZ,
  resolved_at         TIMESTAMPTZ,
  affected_users_count INT,           -- aggregate only; no individual user refs
  root_cause          TEXT,
  impact_summary      TEXT,
  timeline            JSONB[],        -- [{ time, event, detail }]
  action_items        JSONB[],        -- [{ id, title, owner, due, status }]
  responders          TEXT[]          -- staff identifiers only, no child data
}

Rules

  1. TTR definition — Time to Resolve (TTR) is calculated as resolved_at − detected_at. If resolved_at is null, TTR is not shown.
  2. Postmortem complete requirements — Status may only be set to postmortem_complete when all JSONB action items have an owner and due date assigned, and the root cause field is non-empty.
  3. SLA by severity — P0 incidents must have a postmortem opened within 24 hours of resolution. P1 within 72 hours. P2 within 7 days.
  4. No child PII — The incidents table must never contain individual child identifiers, names, session IDs, or any PII. affected_users_count stores aggregated integers only.
  5. Blameless postmortem — Root cause and timeline fields must not reference individual staff names in a blaming context. Responder attribution is stored separately in responders[] for internal audit only and not displayed in the UI.
  6. No session-level data — Incident records track affected session counts as integers. Session-level logs are stored in a separate audit system and are not joined into this view.
  7. Owner-only view — This component and route are strictly Owner-persona gated. Child and parent personas must never receive access to incident data, even in read-only mode.
