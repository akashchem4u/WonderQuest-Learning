Reset Spec
Schema, execution logic, and safety rules for the beta session reset system.
Audit Schema — beta_resets
CREATE TABLE beta_resets ( id UUID PRIMARY KEY, ref_id TEXT, -- RST-NNNN child_id UUID REFERENCES users(id), admin_id UUID REFERENCES users(id), reset_type ENUM('full', 'scenario', 'partial'), scenario_key TEXT NULLABLE, band_scope ENUM('p0','p1','p2','p3') NULLABLE, subject_scope TEXT NULLABLE, reason TEXT NOT NULL, executed_at TIMESTAMPTZ DEFAULT NOW(), -- audit fields ip_hash TEXT, before_snapshot JSONB -- child's XP/stars/progress before reset (for reference) );
Reset Execution — Server-Side Only
-- Full reset: DELETE FROM play_sessions WHERE child_id = :id; DELETE FROM skill_progress WHERE child_id = :id; UPDATE users SET xp=0, streak=0, level=1 WHERE id = :id; -- Scenario reset: -- Full reset (above), then apply scenario fixture (server-side seed data) CALL apply_scenario_fixture(child_id := :id, scenario := :scenario_key); -- Partial reset: DELETE FROM skill_progress WHERE child_id = :id AND band_id = :band AND subject_id = :subject;
Safety Rules
🔒
Admin PIN required
Separate from login password. 4-digit PIN verified server-side before any reset executes.
📷
before_snapshot stored
JSONB snapshot of child's XP, stars, streak, and progress captured before execution. Non-restorable — for debugging reference only.
⏳
Rate limit: max 5 resets per child per day
Enforced server-side. Prevents accidental bulk resets. Returns 429 Too Many Requests if limit exceeded.
✅
is_beta_admin gating
Only users with is_beta_admin = true can execute resets. Enforced at API middleware layer.
📜
Audit log is immutable
No DELETE or UPDATE on beta_resets ever. RLS policy enforces append-only. All resets permanently logged with ip_hash, admin_id, and reason.
🚫
Account data not deleted
Child account (email, profile, settings) is never touched. Only play_sessions, skill_progress, and XP/streak fields on users row.
🛑
Cannot be undone
There is no rollback mechanism. The before_snapshot is advisory only. Contact engineering for urgent database-level recovery.
