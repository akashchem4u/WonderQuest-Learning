Database Schema
-- Class table (beta + production) classes ( id UUID PRIMARY KEY, teacher_id UUID REFERENCES users(id), name TEXT, school_id UUID, grade TEXT, class_code TEXT UNIQUE, -- e.g. 'MR-4829' max_students INT DEFAULT 35, is_beta BOOLEAN DEFAULT false, beta_cohort TEXT, created_at TIMESTAMPTZ DEFAULT NOW() ) -- Pilot schedule per class pilot_schedules ( class_id UUID, weeks_active INT[], -- week numbers 1-6 sessions_per_week INT, session_length_mins INT, feedback_cadence TEXT, pilot_goals TEXT[] )
Class Code Generation
Format: {SCHOOL_CODE}-{4-digit-random}
Unique constraint enforced at DB level; regenerate on collision. Max 10 retries before surfacing error.
School code derived from first two letters of school short-name, uppercased (e.g. "Maple Ridge" → MR).
Teacher may regenerate code at any time before students join. Once a student joins, code is locked.
Student Join Flow
-- Student enters code → look up active class SELECT id FROM classes WHERE class_code = :code AND is_active = true AND student_count < max_students; -- On match → create membership INSERT INTO class_memberships ( class_id, student_id, joined_at ) VALUES ( :class_id, :student_id, NOW() );
Teacher sees student first name only when they join — full name is never displayed in beta dashboard.
Student join is idempotent — rejoining with the same code is a no-op.
Pilot Schedule Rules
Pilot window: 6 weeks, April 7 – May 16, 2025. Teachers select which weeks they'll actively test.
Minimum commitment: 2 weeks of active testing to be included in final cohort analysis.
Feedback cadence stored as 'per_session' or 'weekly'.
Support team uses weeks_active to schedule proactive check-in emails.
This setup flow is distinct from production onboarding — no payment, trial, or subscription steps.
