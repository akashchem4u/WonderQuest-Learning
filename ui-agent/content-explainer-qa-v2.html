⚙ Engineering — Tab 5: Spec Reference This tab is the single source of truth for QA surface behavior, API contracts, hard block rules, and design tokens. Content-admin reviewers can reference this tab to understand enforcement rules. Keep this in sync with the backend ExplainerQAPolicy service.

API Routes

GET /content/explainers/qa-queue
Paginated queue of explainers pending QA. Supports ?status=&band=&page= filters.
GET /content/explainer/:id/review
Full explainer object: script, media hints, rule results, qa_history. Used by Tabs 2–4.
POST /content/explainer/:id/qa-decision
Submit QA verdict. Body: { verdict, override_warnings, override_reason?, notes }. Appends to audit log. Triggers status transition and submitter notification.

Hard Blocks — Must resolve before approval

  • Missing audio transcript — any type==="audio" hint must have a non-empty transcript field. COPPA compliance. No override permitted.
  • Missing correct answer link — explainer must be linked to at least one question node with a defined correct answer.
  • Prohibited content flag — any hint or script segment flagged by automated content moderation with severity HIGH.

Warnings — Reviewer can override

  • Borderline word length — sentence exceeds 8-word soft limit but passes automated grammar check. Reviewer override requires explicit note.
  • Missing image alt text — accessibility best practice. Override allowed but noted in audit log.
  • Single missing media hint — explainer has fewer than recommended hints but at least one. Override allowed if context justifies.

Override Rules

  • Content-admin can override warnings only. Hard blocks always require a fix.
  • Override requires a written reason stored in audit log (override_reason field).
  • Override is surfaced in publish queue with a visible "Approved with override" chip.
  • Hard blocks: Approve with Override button is disabled until all hard blocks are cleared.

QA Decision Flow

  1. Reviewer opens explainer from queue (Tab 1).
  2. Reviews script line by line, checks word limits and tone (Tab 2).
  3. Reviews each media hint, checks transcripts and alt text (Tab 3).
  4. Reviews aggregated blockers and warnings (Tab 4).
  5. Makes verdict: Approve, Return for edit, or Approve with override (warnings only).
  6. Verdict appended to append-only audit log. Submitter notified.

Transcript Requirement

Any audio hint attached to a learner-facing explainer must include a text transcript. This is a hard block and cannot be overridden. The requirement derives from COPPA accessibility provisions and WonderQuest's WCAG 2.1 AA commitment. Transcript must be stored on the hint object (hint.transcript) before the QA surface marks the hint as reviewable.

QA Audit Log

  • Append-only — no mutations after write.
  • Each entry: { reviewer_id, action, timestamp, notes, override_reason? }
  • Surfaced on Tab 4 (last 3 shown; full log accessible via API).
  • Action enum: passed | flagged | blocked | returned | approved | approved_with_override
