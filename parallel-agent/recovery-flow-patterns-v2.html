Recovery Flow Patterns

WonderQuest · Session recovery, offline handling, conflict resolution

recovery-flow-patterns-v2.html — Recovery flow patterns. 4 recovery scenarios: (1) Mid-session disconnect (network drops during play — resume on reconnect), (2) App crash/force-close (resume from last checkpoint on relaunch), (3) Multi-device conflict (same child on two devices — resolve which session wins), (4) Progress sync (local-first writes, sync when online). Key rule: child NEVER loses progress. Stars/XP saved locally first (IndexedDB/localStorage), synced when online. "Your stars are safe!" shown on any recovery. No timer anywhere.

Adult UX principle: No forced resume prompt for parents or teachers. A non-intrusive toast notification appears in the corner: "Session resumed from last checkpoint". Adults stay in control of their workflow — no overlay interruptions.
Behaviour rules
📋 Resume prompt rules
Trigger
App relaunch after crash, OS kill, or explicit close — only if unfinished session exists in IndexedDB
Auto-select
After 8 s of no interaction, Continue is selected automatically. Shown by a filling progress bar — no number countdown.
Start fresh
Previous session XP and stars still preserved — they are never discarded. Child simply begins a new session.
Stars shown
"⭐ 3 stars earned — safely saved!" always displayed in the context card. Reassures child immediately.
Checkpoint
Resume picks up at exact question index written to IndexedDB before crash — last answered Q, not last seen Q.
Desktop adults
Toast only — no overlay. "Session resumed from last checkpoint" disappears after 5 s.

Automatic resolution — if parent takes no action
⏱️
Auto-resolve after 60 s
If no parent action is taken within 60 seconds, the most-advanced session wins automatically (furthest checkpoint, most XP). The other device shows a gentle notification: "Emma's session is continuing on iPad." Parent receives a push notification with a summary.

XP from both sessions is always merged (additive) — no session's progress is discarded.
Conflict resolution rules
⚖️ How conflicts are resolved
XP
Additive — sum XP from both sessions. Child never loses earned XP.
Stars
Take maximum — never reduce star count. Best result from either session wins.
Checkpoint
Take furthest — highest question index survives. Progress never rolls back.
Active session
Most-advanced device wins if no parent resolution within 60 s.
Child experience
Child never sees conflict UI. On the paused device, play simply pauses with a gentle message.
Local-first write order
Write order: 1. Write to IndexedDB (local, immediate — synchronous-feeling to user) 2. Update UI optimistically (child sees stars/XP instantly) 3. Queue sync job (background ServiceWorker or in-memory queue) 4. When online: POST /api/progress/sync 5. Server reconciles with last_written_at timestamp 6. Conflict: server version newer → merge (sum XP, take max checkpoint)

Conflict resolution rules
⚖️ Server-side merge logic
XP
Always additive — sum(xp_earned) from all sessions in window. Never subtract.
Stars
Take maximum — max(stars_earned) across all devices. Stars never decrease.
Checkpoint
Take furthest — max(checkpoint) across sessions. Progress never rolls back.
Active session
Most-advanced session wins if no parent resolution within 60 s.
Reconcile key
last_written_at timestamp — server discards payload if server record is newer, after merging XP/stars.
Recovery guarantees
⭐ Child NEVER loses XP or stars in any recovery scenario
🔒 "Your stars are safe!" shown in every recovery UI
📦 Offline play: up to 20 questions cached per skill
🔁 Re-sync retry: exponential backoff 1s → 2s → 4s → 8s → max 30s

Storage strategy
💾 What lives where
IndexedDB
Full session state — questions answered, XP, stars, checkpoint. Survives app crash, OS kill, and browser close.
localStorage
Last session metadata (skill_id, session_id, timestamp) for fast resume prompt on relaunch.
ServiceWorker cache
Question content (text, images) and cached audio for offline play. Max 20 questions per skill, LRU eviction.
Sync queue
In-memory queue (ServiceWorker) + IndexedDB fallback. Drains on network restoration; retries with exponential backoff.
Server (Postgres)
Source of truth after sync. Reconciles with last_written_at. Merge logic: additive XP, max stars, max checkpoint.
Offline audio strategy
🔊 Audio during disconnect
Cached audio
Up to 20 question audio clips cached per skill in ServiceWorker. Plays without network.
Audio indicator
Speaker icon shows amber dot when using cached audio. No functional difference to child — audio still plays.
Streaming audio
Only live-fetched audio (voice coach, reward sounds) is paused offline. Core question audio always cached.
Pre-fetch strategy
On session start, next 5 question audio clips are pre-fetched in background to ensure offline resilience.
