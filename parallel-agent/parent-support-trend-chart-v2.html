"Support Area" Definition
A subject is classified as a support area when mastery_status = 'Exploring' across ≥2 sessions in the current period, OR when a child has 0 sessions in a subject in the current period. Support areas are NOT defined by low accuracy — accuracy data is never surfaced to parents.
DB Schema
child_support_trends ( child_id UUID NOT NULL, subject_id UUID NOT NULL, week_number INT NOT NULL, -- 1-indexed per academic year engagement_level ENUM( 'exploring', -- just getting started / needs most encouragement 'building', -- gaining momentum 'growing' -- strong progress (would exit support view) ) NOT NULL, sessions_count INT NOT NULL DEFAULT 0, updated_at TIMESTAMPTZ NOT NULL DEFAULT now(), PRIMARY KEY (child_id, subject_id, week_number) ) -- Parallel table to child_strength_trends -- No accuracy columns. No question-level data.

Privacy Rules
No accuracy % — parent view never shows question-level accuracy or "Emma got this wrong".
No question-level data — only subject-level engagement levels are surfaced.
Content gaps framed as system limitation, not child limitation. If a subject has low data due to limited CMS content, surface it as "coming soon", not "Emma hasn't done this".
Suggested Activities Source
All suggested parent activities are sourced from cms_content (updateable by content team). Activities are generic and playful — never prescriptive homework. Language avoids "practice", "worksheet", "test".
Teacher Visibility
Teacher sees class-level support areas (aggregated view). Individual child support data is not in the teacher view per se — teachers have their own separate intervention flow. This chart is parent-only.
