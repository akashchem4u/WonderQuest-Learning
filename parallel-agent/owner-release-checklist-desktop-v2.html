  • targetDate: ISO8601 string — planned deploy date
  • checklistItems: ChecklistItem[] — array from DB query
  • onDeployClick: () => void — fires only when gate passes
  • readOnly?: boolean — hides deploy button for read-only viewers

DB Schema — release_checklist_items

CREATE TABLE release_checklist_items ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), release_version TEXT NOT NULL, category TEXT NOT NULL, item_label TEXT NOT NULL, status TEXT NOT NULL CHECK (status IN ( 'pending','in_progress','complete','blocked','waived' )), owner_role TEXT, completed_by TEXT, completed_at TIMESTAMPTZ, notes TEXT ); -- Index for fast version lookups CREATE INDEX idx_rcl_version ON release_checklist_items (release_version); -- Privacy: no child PII columns in this table

Gate Rules

  • ⛔ ALL items must have status complete or waived before the deploy button unlocks.
  • ⛔ Any item with status blocked prevents deploy regardless of all other items.
  • ⛔ Waiver requires an owner-authored note stored in the notes column. Empty waivers are rejected at the API layer.
  • ⛔ P0 accessibility items cannot be waived. They must reach complete status. (Enforced both client-side and server-side.)

Business Rules

  • BR1 Status transitions are append-only in the audit log. Items cannot be moved backward from complete to pending without an owner override and a mandatory reason note.
  • BR2 Blocked items surface a banner at the top of the checklist tab and are highlighted in the affected category. The deploy button tooltip must reference the blocking item by label.
  • BR3 The overall progress bar counts only complete and waived items as numerator. In-progress, pending, and blocked items are excluded from the count.
  • BR4 Release history records are immutable after deploy. No status changes are permitted to items belonging to a shipped release.
  • BR5 No child PII may appear in any release record field, including notes. The API validates this constraint server-side before persisting any update.
  • BR6 The deploy action triggers a server-side re-validation of all item statuses. If any item is non-terminal at the time of the deploy request, the API returns 409 Conflict and the UI shows an error banner.
