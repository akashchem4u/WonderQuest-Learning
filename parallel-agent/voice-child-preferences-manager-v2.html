Technical Specification

Child-Controlled Settings

  • voice_speed — child may switch Normal ↔ Slow during a session, unless parent has enabled "Lock to slow speed" (lock_speed: true). If locked, the toggle is hidden in the child UI.
  • Replay button ("Hear it again") — always visible and functional; triggers re-read of current question audio asset from beginning.
  • Stop readout — child may tap a stop/skip icon mid-playback to interrupt audio and display text-only mode for the current question. State resets at next question.

Parent-Controlled Settings

  • All other fields (read_aloud, voice_coach, lock_speed, background_music, celebration_sounds, quiet_hours_start, quiet_hours_end) are parent-only on write.
  • Parents can see all current child-set values in the management UI for transparency; child-set preferences are read-only fields in the parent view.
  • The child UI does not expose parent-locked settings as interactive; they render as informational indicators only.

API

  • GET /api/children/:id/voice-prefs — returns full child_audio_prefs record; accessible to parent and child session tokens.
  • PUT /api/children/:id/voice-prefs — updates one or more fields; requires parent auth token. Returns updated record. Child tokens receive 403.
GET /api/children/:id/voice-prefs → { child_id, read_aloud, voice_coach, voice_speed, lock_speed, background_music, celebration_sounds, quiet_hours_start, quiet_hours_end, updated_at } PUT /api/children/:id/voice-prefs (parent auth) Body: { field: value, ... } → 200 updated record | 403 forbidden | 422 validation error

Propagation

  • Voice settings are applied at session start by fetching /api/children/:id/voice-prefs and caching in session state.
  • Parent changes made mid-session apply at the next question boundary; the client polls or listens via websocket event voice_prefs_updated and applies on next question load.
  • Child speed changes (if not locked) apply immediately to the next sentence or utterance without re-fetching from API — client-local state only for that session.

Quiet Hours Enforcement

  • Client-side: evaluated before each audio playback call.
  • Server-side: audio asset requests check quiet hours and respond 204 (no content) if within window, preventing workarounds.
// Client-side quiet hours check function shouldSuppressAudio(prefs) { const now = new Date(); const h = now.getHours() * 60 + now.getMinutes(); const start = parseTime(prefs.quiet_hours_start); // e.g. 21:00 → 1260 const end = parseTime(prefs.quiet_hours_end); // e.g. 07:00 → 420 // window wraps midnight if (start > end) return h >= start || h <= end; return h >= start && h <= end; } if (shouldSuppressAudio(prefs)) suppressAudio();

Database

  • Table: child_audio_prefs — one row per child, keyed on child_id (FK to children table). All audio/voice fields stored in this table. No separate voice-specific table.
  • Schema fields: child_id, read_aloud BOOL, voice_coach VARCHAR, voice_speed ENUM(normal,slow), lock_speed BOOL, background_music BOOL, celebration_sounds BOOL, quiet_hours_start TIME, quiet_hours_end TIME, updated_at TIMESTAMP.

Accessibility (WCAG)

  • All toggle inputs use <label for="..."> with matching id on the <input>.
  • Toggle state communicated via aria-checked on the <input type="checkbox">; updated dynamically on change.
  • Parent-lock badges linked via aria-describedby on the toggle input referencing the badge element ID.
  • Summary chip row uses aria-live="polite" so screen readers announce changes after parent toggles.
  • Child switcher buttons use aria-pressed to indicate active selection.
