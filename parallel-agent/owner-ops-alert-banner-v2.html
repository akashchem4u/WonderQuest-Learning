ENG NOTE — owner-ops-alert-banner-v2.html
Contextual alert banners rendered at the top of the owner console content area (below KPI strip, above page content). Multiple banners can stack.
Severities: P0 (red, non-dismissible) · P1 (amber, dismissible) · Info/Release (green, dismissible) · Privacy (violet, requires acknowledgement).
Trigger sources: Route monitoring (P0/P1) · Feedback SLA breach (P1) · Release gate change (Info) · Privacy action required (Privacy) · Content block (P1).
Stacking: P0 banners always on top. Then P1. Then Info. Max 3 banners shown; "+N more" chip if additional alerts exist.
Dismissal: P0 = non-dismissible until resolved. P1 = dismissible (owner action: "View" or "×"). Info = dismissible. Privacy = requires "Acknowledge" click (not simple ×).

Ops alert banner stack. 4 severity types. Max 2 banners shown, +N overflow chip. P0 non-dismissible. Privacy requires acknowledgement. Stack order: P0 → P1 → Info/Privacy.

Stacking Rules

Maximum 2 banners shown simultaneously (prevents banner fatigue). Priority order: P0 always first, P1 second, then Info/Privacy by timestamp (newest first). If >2 active banners, a "+N more" overflow chip is shown below the 2nd banner. Clicking overflow chip opens a modal list of all active alerts.

Persistence

Dismissed banners (×) are stored in owner_dismissed_alerts table with timestamp. They do not re-appear unless the underlying condition changes (e.g., a new feedback SLA breach creates a new alert). P0 banners are never stored as dismissed — they re-render on page load until resolved. Privacy acknowledgement is stored in privacy_acknowledgements table (governance log entry created).

Banner on Mobile

Mobile shows the same banners but with a more compact layout (icon + title only, sub-text collapsed behind a "Details" tap). P0 banners on mobile are also shown as a dedicated non-dismissible card at the top of the home screen feed. P0 is also triggered as a push notification (FCM/APNs) when first declared.
