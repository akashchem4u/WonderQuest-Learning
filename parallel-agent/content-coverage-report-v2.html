Coverage Calculation

Skills are counted as "covered" if their review_status is Published or Under Review. Coverage is computed per band + subject cell.

SELECT band_id, subject_id, COUNT(*) AS covered_skills FROM content_skills WHERE review_status IN ('Published', 'Under Review') GROUP BY band_id, subject_id;
v2.6 Target Store

Per-cell targets are stored in a versioned targets table, enabling comparison across releases.

content_targets( band_id TEXT, subject_id TEXT, target_skills INTEGER, target_questions INTEGER, version TEXT -- e.g. 'v2.6' )
Gap Detection

Gap = target_skills - current_skills for each cell. Severity thresholds:

SeverityConditionAction
Criticalgap > 5Immediate commission, sprint assignment
Highgap 3–5Commission within current sprint
Mediumgap 1–2Backlog, next sprint planning
Commission Flow

Commissioning a gap creates a record in content_commissions and fires a webhook to sync with Notion and Linear.

content_commissions( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), band_id TEXT NOT NULL, subject_id TEXT NOT NULL, topic_name TEXT NOT NULL, priority TEXT CHECK (priority IN ('Critical','High','Medium')), target_count INTEGER NOT NULL, commissioned_at TIMESTAMPTZ DEFAULT now(), sprint_id TEXT, -- linked sprint/cycle notion_page_id TEXT, -- synced on webhook linear_issue_id TEXT -- synced on webhook );

Webhook payload includes band_id, subject_id, topic_name, priority, target_count, and commissioned_at.

Data Refresh

Coverage snapshots are generated daily via a scheduled job. Real-time counts are available by querying SELECT COUNT(*) directly on content_skills. The snapshot date is stored in coverage_snapshots(snapshot_date, band_id, subject_id, covered_skills, total_questions) for historical trending.
