Workflow Spec & Schema beta-feedback-resolution-loop-v2
Database Schema SQL 
-- Lifecycle event log
CREATE TABLE feedback_loop_events (
  id           UUID         PRIMARY KEY,
  feedback_id  UUID         REFERENCES feedback(id),
  event_type   ENUM(
                 'submitted', 'triaged', 'in_review',
                 'action_taken', 'loop_closed', 'tester_responded'
               ),
  event_data   JSONB,
  actor_id     UUID         NULLABLE,  -- null for auto-triage
  created_at   TIMESTAMPTZ  DEFAULT NOW()
);

-- Tester notification records
CREATE TABLE loop_notifications (
  id              UUID,
  feedback_id     UUID,
  message         TEXT,
  sent_at         TIMESTAMPTZ  NULLABLE,
  opened_at       TIMESTAMPTZ  NULLABLE,
  response_rating INT          NULLABLE,   -- 1-5 stars
  response_text   TEXT         NULLABLE
);

-- Notification templates (i18n-ready)
CREATE TABLE notification_templates (
  key           TEXT,
  locale        TEXT,           -- e.g. 'en-US'
  body_template TEXT            -- vars: {{issue_id}} {{resolution_summary}} {{version}}
);
SLA Tracking Rules Rules
Priority Threshold Measurement Overdue condition
P0 < 4 hours action_taken from submitted elapsed > 4h AND status NOT IN resolved / wont_fix / deferred
P1 < 24 hours action_taken from submitted elapsed > 24h AND status open
P2 < 72 hours action_taken from submitted elapsed > 72h AND status open
P3 < 1 week Best effort No hard SLA — tracked for reporting only
Feature 2 sprints Triage decision within 2 weeks of submission Unactioned after 2 sprints = overdue 
-- Overdue query example
SELECT f.id, f.priority, f.submitted_at,
       NOW() - f.submitted_at AS elapsed
FROM   feedback f
WHERE  f.status NOT IN ('resolved', 'wont_fix', 'deferred')
  AND  (
         (f.priority = 'P0' AND NOW() - f.submitted_at > INTERVAL '4 hours')
    OR   (f.priority = 'P1' AND NOW() - f.submitted_at > INTERVAL '24 hours')
    OR   (f.priority = 'P2' AND NOW() - f.submitted_at > INTERVAL '72 hours')
       );
Tester Notification & Privacy Rules Privacy
🔔
Delivery channel: In-app notification only
Notifications are delivered exclusively via the WonderQuest in-app notification center. Email delivery is disabled for all beta tester notifications to protect tester privacy and avoid unsolicited external contact.
🏫
Tester identity: School/cohort label only in owner view
Personal names of testers are never shown in the owner feedback dashboard. The owner sees only the tester's school label (e.g., "Maple Ridge Elementary") or cohort label. This prevents inadvertent privacy exposure and reduces bias in triage.
📝
Template variables: {{issue_id}} · {{resolution_summary}} · {{version}}
All notification messages are rendered from templates stored in notification_templates(key, locale, body_template). Templates support locale-aware rendering. Owner can edit the message before sending.
↩️
Tester response: Optional 1–5 star rating + free text
Testers may respond to the loop-close notification with a star rating and optional comment. Response is stored in loop_notifications.response_rating and .response_text. Response is never required.
