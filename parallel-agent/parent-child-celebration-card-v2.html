Business Rules
Rule 1 — First Name Only
The card renders the child's first name exclusively. Full name, username, or child ID must never appear on a shareable card surface.
Rule 2 — No Accuracy or Score Display
No percentage scores, correct/incorrect counts, or any metric that could signal a learning deficit may appear on or adjacent to the celebration card at any time.
Rule 3 — Zero Cross-Child Comparison
The celebration history view shows only the current child's milestones. Sibling data, class averages, or any peer comparison is strictly prohibited in this component.
Rule 4 — Additive Framing Only
All copy must frame achievements as gains ("collected", "achieved", "explored"). Negative frames ("missed", "lost", "fell behind", "reset") are never permitted in celebration surfaces.
Rule 5 — Server-Side Celebration Trigger
Celebration records are inserted by a server-side database trigger or background job when milestone conditions are met. The client must never write to child_celebrations directly; it only reads.
Rule 6 — card_viewed_at Logged Once
On first card load, the server sets card_viewed_at via a PATCH /celebrations/:id/viewed endpoint (idempotent — subsequent calls no-op). This field is used for engagement analytics only and is never surfaced to the parent or child.
